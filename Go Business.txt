Go Business
Referral Dashboard — Coding Assessment

1. Application Overview
This document outlines the requirements and flow for the Referral Dashboard Application, a web-based productivity tool built for Go Business to help users track referrals, earnings, and partner activity through a structured dashboard.
Objective
To build a secure, responsive, and intuitive referral management system featuring user authentication, a referrals overview pulled from an API, a searchable, sortable, and paginated referrals table, referral link and code sharing, and a referral detail page.
2. Application Flow
[ User Authentication ] ──> [ Protected Dashboard ] ──> [ Referral Management ]
                                                                │
                                   [ Logout ] <─────────────────┘
User Authentication: Users must log in via a secure endpoint using their email and password.
Dashboard Access: Upon successful authentication, users are redirected to the protected referral dashboard at /.
Referral Management: Users can view overview metrics, service summary details, their referral link/code, and search and sort their referrals via the API, with the results paginated client-side at 10 rows per page.
Logout: Users can securely log out, which clears the session cookie and returns them to the login screen.
Routing Notes
Unauthenticated users attempting to access a protected route (/ or /referral/:id) are redirected to /login.
Authenticated users visiting /login are redirected to /.
The Not Found route (*) must remain reachable without a token and must not be wrapped in ProtectedRoute.
App.jsx must wrap <Routes> in <BrowserRouter> so the whole app works when the portal only loads App.jsx; main.jsx should render <App /> only, with no router there.
3. Authentication API
Login Endpoint
Method: POST
URL: https://v9fes04dwf.execute-api.eu-north-1.amazonaws.com/api/auth/signin
Request Payload
{
  "email": "<user_email>",
  "password": "<user_password>"
}
Success Response
{
  "data": {
    "token": "<jwt_token_string>"
  }
}
What happens on success: The token is read from responseJson.data.token (nested under data) and saved with Cookies.set('jwt_token', token). The application then navigates the user to /.
Failure Response (401)
{
  "message": "Invalid email or password"
}
What happens on failure: The message is read from responseJson.message and the error text "Invalid email or password" is displayed on the login page. The Sign in button stays enabled even when fields are empty — every click performs the POST to the login API, so empty fields, partial fields (email only or password only), or wrong credentials all receive the API error response and the user remains on /login.

Referral Details Page (/referral/:id)
Accessible by clicking on a specific row in the referrals table.

6. Technical Implementation Details
Cookie Implementation
Auth Token: Stored cookie name: jwt_token (set via Cookies.set('jwt_token', token) on login success; removed on logout)
Protected Route Logic
ProtectedRoute checks for the cookie using Cookies.get('jwt_token')
If the cookie exists, render the component
If the cookie is missing, redirect automatically to /login
The * (Not Found) route must not be wrapped in ProtectedRoute
Authentication Flow (How the Token Works)
There is no separate, fixed API key for the dashboard or referrals data. The only credential the candidate needs is the test email and password from Section 7, used once against the login endpoint.
Step 1 — Sign in: POST the email/password to the login endpoint (Section 3). The response contains a JWT at data.token.
Step 2 — Store the token: Save that JWT in the jwt_token cookie (Cookies.set('jwt_token', token)). This token is the only credential used for every subsequent request.
Step 3 — Reuse the token: For both the referrals list and referral detail requests, read the token back out of the cookie (Cookies.get('jwt_token')) and send it as Authorization: Bearer <jwt_token>. There is nothing else to configure — no API key, no separate secret.
If the token is missing or expired, the referrals API will reject the request; in that case, treat it the same as any other failed fetch (show the error message/status, per the Dashboard Page loading & error behavior above).
Referrals API Logic
All requests below use the same base URL with different query parameters, and all require the Bearer token described above.
Purpose
Request
Full list (default)
GET /api/referrals
Search by name or service
GET /api/referrals?search=pm  (also accepts ?q=graphics)
Single referral by id
GET /api/referrals?id=5
Sort by date
GET /api/referrals?sort=asc  (asc = oldest first, desc = newest first; default is desc)


Base URL for all of the above: https://v9fes04dwf.execute-api.eu-north-1.amazonaws.com/api/referrals — keep the path and host exactly as written so any shared mock or gateway matches every call.
search and q are interchangeable; either query parameter filters by name or service name. Send whichever the candidate's UI logic produces, but only one is needed.
sort accepts asc or desc; if omitted, the API defaults to desc (newest first), matching the UI's default sort option.
search/q and sort can be combined in the same request (e.g. ?search=pm&sort=asc) to filter and sort together.
The API does not page the response — every call above returns the complete matching set. Pagination (10 rows per page) must be implemented client-side on whatever array the API returns.
Referral detail: GET the same base URL with ?id=<referralId> and the same Authorization header.
